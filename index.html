发表于 2020-03-08 | 分类于 Android手机破解相关

简介

关于国行N9108V安卓6.0我参照https://www.cnblogs.com/jsp1256/p/7536579.html三星S5_G9008V 解锁成功，通过解析文章的思路，大概跟早期4.4时期破解思路类似，只不过后期更新关掉了手机端直接编辑相关文件的权限，但是通过这个思路就是用过修改EFS文件来实现破解，而且理论上刷机不会刷写EFS分区，达成永久破解。
实际上早期部分手机，尤其是三星旗舰机支持双网的4g，但是在配置文件内软屏蔽，像现在的三网随意已经没有这种事情了。其实这种事情主要存在于三网来临的前夕，硬件满足但各营运商又想去保护自己的用户不想让丫用其他营运商卡的时代。

原文章链接 http://bbs.gfan.com/android-7908992-1-1.html。

解决问题

1.在拨号键盘输入*#0808#，选择DM+MODEM+ADB模式。（破解完后记得改回去）（小白注：改回去选第一个，MTP即可，第二个MTP+ADB会开了你的USB调试）
2.在我的电脑-》右键-》属性-》设备管理-》端口下记住三星usb链接的com号
若没有说明驱动安装失败。
3.打开高通QPST的连接配置工具QPST comfiguration，并手动添加端口（小白注：就是上面的COM号）
添加完成后，会在QPST Configuration中看到端口处于连接状态添加完成后，会在QPST Configuration中看到端口处于连接状态
4.打开EFS Explorer,选择MSM8974对应的端口，点击ok，等待一会
5.EFS Explorer与手机连接成功后，在左侧目录找到policyman文件夹，在carrier_policy.xml文件那右键先copy date file to pc把文件复制到电脑，再->dalete删除，空白处右键->copy date file from pc
点击browe,选择刚刚下载的carrier_policy.xml，ok

然后重启！

记一次修复用Java汉字转拼音的开源代码的坑

发表于 2020-01-15 | 分类于 Android开发中遇到的问题

遇到问题

有这么一个需求，把汉字转为拼音。首先分析这个需求，猜想网上肯定很多现成的类库，而且印象里有象Pinyin4j这样的库很多，甚至想过要加入字典来实现，后来经过考量，首先的话这是一个很小的需求，并非主要业务，其次的话这个转换后的拼音可以用户手动纠正，也就是说这只是个辅助输入的功能。
输入部分就不说了，很简单给textview加个监听器，当输入完成后去读取然后转换。经过多方查询，了解到有这么一个HanziToPinyin的类，这个是从早期的Android代码里提取出来的，源自4.2的代码，并且采用的人数还不少。根据之前的代码已经解决了关于地区的问题。
尝试引入，这种类的引入很简单，把代码cv过来就行了。（代码贴在后面）测试一下。

在5.1.1的系统上测试没问题，提测之后遇到了问题，在测试得8.0以上的系统中，大概在拼音p之前的转换成功，但之后的字会转换失败。

分析问题

分析问题就要分析代码了，之前有看过原理，大概原理就是这样子的。
这个类里面维护了两个数组，一个是根据所有拼音得开头分割点得汉字得unicode的数组。例如a的开头的第一个字是啊，对应的“\u554a”这类得，另一个数组维护了这个字到拼音的映射的ASCLL数组。例如a是[‘26’,’0’,’0’,’0’,’0’,’0’]。然后根据传入的字进行二分查找，利用java自带的Collator的compare方法返回两个汉字的顺寻进行查找。查找效率很高。
但很诡异的是在旧版本的系统上这个运行没有问题，在8.0之后产生了问题，第一个猜想是因为8.0的String.getBytes()返回有问题，但全局查找没有用到，从此处怀疑可能是编码有变化。
后来经过各种姿势的查证，并非编码变化。问题，最后将希望寄居在对比过程。
用log打出同一个字的对比过程，将正确与错误的流程进行对比，发现在目标字与“狅”（狂的错字版本）对比的时候，正确的版本返回了错误的结果，例如“你”cmp”狅”时，应该是返回-1，因为n在k的后面，但结果却返回了1，经过对字表的查证，发现“狅”的位置应该是对应“qing”的读音。破案了。

解决问题

找打了问题来源，就好解决了，把狅替换为“qing”读音开头的第一个字，例如“青”的unicode码。
修复了这个问题，而这个问题在之前正确的本质应该是错和错对应了对。以毒攻毒，这应该也是之前旧版本的cmp方法存在的问题。

一次因格式化价格引发的问题

发表于 2020-01-09 | 分类于 Android开发中遇到的问题

遇到问题

开发任务不是很多的时候日常看一下听云，注意到一个很不扎眼的崩溃统计，注意到这个崩溃几乎只发生在几个人的设备上，占比也不高，很诡异的数组角标越界。

特点描述已经到位了，与后台沟通，但这个确实很难复现。

分析问题

定位问题是最重要的，由于听云的统计信息是混淆过的，但根据混淆代码的mapping文件可以定位到代码出错的位置。

是根据后台返回的一个double价格（这里要说一下，涉及金融相关的内容不要使用double会有问题，但只是做价格展示，并且代码耦合度比较高，暂时不考虑重构，只是为了解决问题。）

这部分代码不是我写的，于是先分析代码。

根据统计信息，可以定位到“if (“00”.equals(d[1])) {”这一行，显示为数组角标越界，长度1引用1，也就是说d[]这个数组的长度不对，很可能是前面的split方法没有成功根据.来分割出两个字符串。

可理论上DecimalFormat(“#.00”)这个方法会根据后面的格式来转换成 x.x的形式啊，理论上不会出错。猜测问题出在前三行，抽取对应方法写一个测试的demo，首先猜测是0之类的问题导致的这个问题，但测试了许多数字之后发现并非如此。于是决定追踪DecimalFormat的源码。于是注意到当DecimalFormatSymbols()为空构造时会默认使用系统的语言构造。

系统默认的语言构造？结合极少设备发生这个问题，又是用.来split，于是着手与查询相关国家的标准问题。当我把手机地区调整为德国时，我所写的测试app会闪退，同时注意到会抛出异常。原数字被转化为“，”分割的了，突然想起来德国的数字分割方式.与，是相反的。

解决问题

找到问题的根源就好解决了。
可以说手动
df.setDecimalFormatSymbols(DecimalFormatSymbols.getInstance(Locale.CHINA));
也可以
df.setDecimalSeparator('.');
但这种手动指定解码区域的方式显然不可取，并不推荐，但确实可以解决问题。在年前本着尽量少改动以免年都过不好的精神目前先这么解决。
等年后考虑对这部分进行重构。

探索优化扫描PDF417码的时候遇到的坑

发表于 2020-01-09 | 分类于 Android开发中遇到的问题

遇到问题

终所周知的，app里常见的扫码功能一般是用于扫描二维码，所以我们直接集成了好用的ZXING库，但突然我们加了一个需求，是需要扫描PDF417码，就是长条形的二维码。

常用的码有很多协议，比较常见的就是一维码，一维码又分为很多协议，我们不做讨论，二维码分为矩阵码，常见的QRCode 和我这次遇到的PDF417 码，经过百度多方查找，确信了ZXING本身支持PDF417码的扫描，所以问题就集中在我集成的版本上了。

经过测试，是可以经过比较久的时间后扫描出内容的，但内容似乎与码内容不符。如果不看解析结果很可能被虚假的解析所迷惑。

分析问题

首先分析问题，分析是在哪个流程出现的问题，扫码的流程分为码定位，码识别，码解析。而前面两个都是由库完成的，因为其他码识别没问题，可以跳过其他步骤。

于是怀疑是解析步骤问题，例如没有指定解析类别。

跟踪代码，查看解析结果，发现问题在于扫描的虽然慢，但扫描却是有结果，于是在解析步骤后打印Log，注意到解析类别输出的不对劲识别类别为“UTC—E”，经过查询，这类码属于一维码。本来打算追踪源码，但需求修改产品催的比较急，于是着手查找解决方案。

解决问题

于是顺代码的回调查找到DecodeUtils，最开始追踪错误，走到了DecodeImgUtils里面，加入猜测的Type为PDF417后继续扫码无效果。在DecodeUtils里面的优先解码器选择内加入PDF417解码，测试扫码，问题解决。

这个问题的痛点在于，之前能够扫描出来码，很容易被迷惑，但实际上是将二维码错误的降级识别为模糊的一维码，导致虽然解码成功，但实际上走了错误的方式。添加正确的解码方式后问题即解决。
